## Purpose

This document defines the Phase 1 product format for reflecting an Adeks-calculated discount into Selcafe through a cashier-entered `kasaislem` row.

The goal is to support the K-OS-008 settlement green-light without placing Adeks customer, coupon, member, or bill identifiers into Selcafe.

## Source Context

The 2026-06-30 kasaislem mechanism QA corrected the earlier fixed-format wording.

Corrected mechanism:

- Settlement truth is `adisyon.toplam_tutar`: full, pre-cash-discount, fixed after settlement.
- `adisyon.kasaislem_no` is a stub and must not be used as the bill↔till link.
- The Adeks discount is reflected as a manually entered `kasaislem` row.
- The row is selected by a dedicated Adeks `islem_tip` plus a pseudorandom one-time code in `aciklama`.
- The discount amount is entered as a positive credit in `alacak`.
- Adeks holds the internal `code → adisyon_no → expected discount` mapping.
- The Selcafe row carries no `adisyon_no` and no Adeks customer/coupon/member id.
- Green-light comparison: `adisyon.toplam_tutar − kasaislem.alacak` compared with Adeks's discount-inclusive total, within 2%, fail-closed.

## Product Terms

| Term | Meaning |
|---|---|
| `fiş / fiş numarası` | Selcafe/cashier artifact. It may identify the Selcafe bill/addition inside cashier operations, but it is not the Customer PWA linking key. The app link is the desk-side, one-time QR handshake under K-OS-009. |
| `adisyon.toplam_tutar` | Selcafe settlement truth: full, pre-cash-discount, fixed after settlement. |
| Dedicated Adeks `islem_tip` | A Selcafe transaction-type entry created once in the Selcafe admin panel and selected by the cashier from the Kasiyer dropdown when reflecting an Adeks discount. |
| Adeks discount code | A pseudorandom one-time code generated by Adeks for one discount issuance. |
| `kasaislem.aciklama` | Free-text Selcafe field that carries the Adeks discount code using the exact parseable format in this spec. |
| `kasaislem.alacak` | Positive credit amount used for the Adeks discount reflection. |
| Green-light | Cashier-facing settlement confirmation that the Selcafe settled amount after the reflected discount matches Adeks's discount-inclusive total within the approved 2% threshold. |

## Pilot Setup Prerequisite — Dedicated Adeks `islem_tip`

Before discount reflection can operate in the pilot, Selcafe must have exactly one dedicated transaction type for Adeks discount reflection.

| Requirement | Rule |
|---|---|
| Admin-panel setup | A dedicated Selcafe `islemtip` entry must exist before pilot use. |
| Cashier-visible label | `ADEKS INDIRIM` |
| Adeks internal alias | `ADEKS_DISCOUNT` |
| Cashier action | Cashier selects this value from the Selcafe Kasiyer dropdown when entering the discount reflection row. |
| Numeric id | The Selcafe numeric `islem_tip` id is determined by Selcafe after admin-panel setup and must be recorded in Adeks configuration after Pod B review. Product docs must not assume or hard-code the numeric id. |
| Failure behavior | If the dedicated `islem_tip` entry does not exist, is duplicated, is renamed ambiguously, or cannot be selected by the cashier, Adeks discount reflection is not operational and the settlement green-light must remain unavailable. |
| Label encoding decision | [RESOLVED - POD B CONFIRMED] The Selcafe-visible label is `ADEKS INDIRIM` (ASCII). The label is cosmetic; Adeks matches the discount reflection row by the numeric dedicated `islem_tip` id plus the pseudorandom one-time code. ASCII vs dotted-I does not affect correctness. The internal Adeks alias remains `ADEKS_DISCOUNT`. |

## Code Generation Requirements

Adeks must generate a pseudorandom one-time code for each discount issuance.

| Requirement | Rule |
|---|---|
| Code type | Pseudorandom, not enumerated or sequential. |
| Random source | Cryptographically secure random generation. |
| One-time use | One code is valid for one discount issuance only. |
| Reuse | A code must never be reused, including after cancellation, cashier mistake, timeout, or correction. |
| Charset | `23456789ABCDEFGHJKMNPQRSTUVWXYZ` |
| Excluded characters | `0`, `1`, `I`, `L`, `O` are excluded to reduce cashier transcription errors. |
| Random payload length | 16 characters from the charset, displayed as four groups of four. |
| Entropy | At least ~79 bits from the random payload. |
| Display format | `DSC-XXXX-XXXX-XXXX-XXXX` |
| Synthetic example | `DSC-7Q2X-K9MD-3T8P-AV6F` |
| Adeks internal mapping | Adeks records `code → adisyon_no → expected discount amount → issuance status` internally. |
| Selcafe exposure | Selcafe receives only the code, not the internal mapping. |

## `aciklama` Embedding and Delimiter Rule

The cashier-entered `kasaislem.aciklama` value must be exactly parseable.

### Required `aciklama` format

```text
ADEKS_DISC|v1|DSC-XXXX-XXXX-XXXX-XXXX|
```

Synthetic example:

```text
ADEKS_DISC|v1|DSC-7Q2X-K9MD-3T8P-AV6F|
```

### Parsing rule

Adeks must treat the `aciklama` value as valid only if the trimmed full field exactly matches this pattern:

```text
^ADEKS_DISC\|v1\|DSC-[23456789ABCDEFGHJKMNPQRSTUVWXYZ]{4}-[23456789ABCDEFGHJKMNPQRSTUVWXYZ]{4}-[23456789ABCDEFGHJKMNPQRSTUVWXYZ]{4}-[23456789ABCDEFGHJKMNPQRSTUVWXYZ]{4}\|$
```

### Delimiter rule

| Requirement | Rule |
|---|---|
| Full-field match | The full trimmed `aciklama` field must match the required format. |
| No substring matching | Adeks must not accept a code found only as a substring inside other free text. |
| No extra notes | Cashier must not add customer names, table notes, coupon names, member identifiers, phone numbers, `adisyon_no`, or other text to `aciklama`. |
| Single code | Exactly one code is allowed per row. |
| Version marker | `v1` is included so future format changes can be separated without ambiguity. |
| Collision handling | If more than one `kasaislem` row matches the same dedicated `islem_tip` + code, the result is not clean and must fail closed. |

## Selcafe Row Content

The cashier-entered Selcafe `kasaislem` row must carry only the minimum fields required for discount reflection and later verification.

| Field | Product rule |
|---|---|
| `islem_tip` | Dedicated Adeks discount transaction type selected from the Kasiyer dropdown. |
| `aciklama` | Exact `ADEKS_DISC\|v1\|DSC-XXXX-XXXX-XXXX-XXXX\|` value. |
| `alacak` | Positive discount amount. |
| `borc` | Not used for the Adeks discount reflection. |
| `adisyon_no` | Must not be entered into the Selcafe row. |
| Adeks customer id | Must not be entered into the Selcafe row. |
| Adeks coupon id | Must not be entered into the Selcafe row. |
| Adeks member id | Must not be entered into the Selcafe row. |
| Phone/name/table notes | Must not be entered into the Selcafe row. |

| Requirement | Rule |
|---|---|
| Amount field | `kasaislem.alacak` |
| Sign | Positive credit only. |
| Meaning | The Adeks discount amount to subtract from `adisyon.toplam_tutar` for the green-light comparison. |
| Zero amount | Not valid for discount reflection. |
| Negative amount | Not valid for discount reflection. |

## Green-Light Product Rule

At settlement, Adeks performs the following product-level check:

1. Read the QR-linked session's Selcafe bill/addition.
2. Read `adisyon.toplam_tutar` for that bill/addition.
3. Read the matching `kasaislem` discount row by:
   - dedicated Adeks `islem_tip`; and
   - exact `aciklama` code match.
4. Join those reads inside Adeks using Adeks's internal `code → adisyon_no → expected discount` mapping.
5. Compare:

```text
adisyon.toplam_tutar − kasaislem.alacak
```

against:

```text
Adeks discount-inclusive total
```

6. Green-light only when the result is within the approved 2% threshold.

Selcafe remains the settlement source of truth. The `kasaislem` row is a verification/control row for discount reflection, not settlement truth.

## Fail-Closed Cases

Adeks must not green-light settlement if any of the following occurs:

| Case | Required behavior |
|---|---|
| No matching dedicated Adeks `islem_tip` row | No green light; manual check. |
| No matching code in `aciklama` | No green light; manual check. |
| `aciklama` contains the code as a substring but not in the exact full-field format | No green light; manual check. |
| More than one row matches the same `islem_tip` + code | No green light; manual check. |
| Code exists in Selcafe but was never issued by Adeks | No green light; manual check. |
| Code was issued by Adeks but maps internally to a different `adisyon_no` | No green light; manual check. |
| `alacak` is missing, zero, negative, or not parseable as a valid money amount | No green light; manual check. |
| `borc` is used instead of `alacak` | No green light; manual check. |
| Difference exceeds 2% threshold | No green light; manual check. |
| Dedicated Adeks `islem_tip` setup is absent, duplicated, or ambiguous | No green light; manual check. |
| Legal/KVKK or ADR-005 read-surface gate is not cleared | No green light feature may be implemented. |

## Synthetic Example

Scenario:

| Field | Synthetic value |
|---|---|
| `fiş` | `100045` |
| Adeks discount code | `DSC-7Q2X-K9MD-3T8P-AV6F` |
| `kasaislem.aciklama` | `ADEKS_DISC\|v1\|DSC-7Q2X-K9MD-3T8P-AV6F\|` |
| Dedicated `islem_tip` label | `ADEKS INDIRIM` |
| Discount amount | `₺25.00` |
| `kasaislem.alacak` | `25.00` |

Adeks internal mapping:

```text
DSC-7Q2X-K9MD-3T8P-AV6F → fiş/adisyon 100045 → expected discount ₺25.00
```

Selcafe row must not contain:

- `100045` in `aciklama`;
- Adeks customer id;
- Adeks coupon id;
- Adeks member id;
- customer name;
- phone number.
